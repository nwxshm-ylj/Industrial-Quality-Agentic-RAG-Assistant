# 一、软件工程与交付流程

## 1. 幂等（Idempotency）

### 定义

同一个操作执行一次和执行多次，系统最终状态相同。

```text
设置订单状态为“已取消” → 幂等
订单金额增加 100 元       → 非幂等
```

### 常见场景

- 支付、退款、订单创建
- 消息队列重复消费
- 网络超时后的请求重试
- API 重复提交

### 常见实现方式

1. **幂等键**：每次业务操作携带唯一 `Idempotency-Key`。
2. **数据库唯一约束**：防止同一业务编号重复写入。
3. **状态机判断**：订单已支付时，不再重复扣款。
4. **请求结果缓存**：相同幂等键直接返回第一次结果。

### 核心结论

> 幂等不是“只能执行一次”，而是“即使重复执行，也不会产生额外副作用”。

## 2. 冒烟测试（Smoke Testing）

### 定义

冒烟测试是一种快速、浅层测试，用于确认系统核心功能是否可运行，是否值得继续做深入测试。

### FastAPI 项目常见检查

- 应用能否启动
- `/health` 是否返回 `200`
- 数据库是否可连接
- 用户是否能登录
- 核心接口是否能返回结果
- 部署后的容器是否存活

```python
from fastapi.testclient import TestClient
from app.main import app

client = TestClient(app)


def test_health():
    response = client.get("/health")
    assert response.status_code == 200
```

### 核心结论

> 冒烟测试不能证明系统没有缺陷，只能证明系统没有严重到无法继续测试的问题。

## 3. CI/CD 是什么

### CI：持续集成

代码提交后自动完成：

```text
代码检查 → 单元测试 → 集成测试 → 构建验证
```

目的：尽早发现代码问题，防止有问题的代码合并进主分支。

### CD：持续交付 / 持续部署

- **持续交付**：代码已具备发布条件，但生产发布通常需要人工审批。
- **持续部署**：所有检查通过后，自动发布到生产环境。

### 典型流程

```text
提交代码
→ 创建 Pull Request
→ Ruff / MyPy / Pytest
→ Docker Build
→ 合并 main
→ 构建正式镜像
→ 部署测试环境
→ 冒烟测试
→ 生产审批
→ 灰度或滚动发布
→ 监控和回滚
```

## 4. GitHub 能否实现 Pipeline

可以。GitHub 通过 **GitHub Actions** 实现 CI/CD。

目录结构：

```text
.github/
└── workflows/
    ├── ci.yml
    └── deploy.yml
```

基础 CI 示例：

```yaml
name: Python CI

on:
  pull_request:
    branches: [main]
  push:
    branches: [main, "feature/**"]

jobs:
  test:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-python@v5
        with:
          python-version: "3.11"

      - name: Install dependencies
        run: |
          pip install -r requirements.txt
          pip install pytest ruff

      - name: Lint
        run: ruff check .

      - name: Test
        run: pytest -v

      - name: Docker build
        run: docker build -t app:${{ github.sha }} .
```

### GitHub Actions 可实现

- 自动测试
- Docker 镜像构建
- 推送至 GHCR
- SSH 部署服务器
- Environment 审批
- Secrets 管理
- 部署后冒烟测试

## 5. 大企业 Python CI/CD 的特点

大企业流程不只是“测试通过后部署”，还包含治理和审计。

### Pull Request 阶段

```text
Ruff / Black
→ MyPy
→ 单元测试
→ 集成测试
→ 覆盖率检查
→ SAST 安全扫描
→ 依赖漏洞扫描
→ Docker Build
→ Code Review
```

### 合并主分支后

```text
构建不可变制品
→ 镜像扫描
→ 生成 SBOM
→ 推送制品仓库
→ 部署 Dev / Test / Staging
→ 自动化验收
→ 生产审批
→ 灰度发布
→ 监控与回滚
```

### 核心原则

- 主分支受保护，不能直接 Push。
- 测试失败不能合并。
- 构建一次，同一制品逐级晋级。
- 使用版本号或 Commit SHA，不只使用 `latest`。
- 生产 Secret 不写进代码仓库。
- 部署后必须进行健康检查和监控。

## 6. Alembic

### 定义

Alembic 是 SQLAlchemy 项目常用的数据库迁移工具，用于管理数据库表结构版本。

### SQLAlchemy 与 Alembic 的区别

```text
SQLAlchemy：定义模型、查询和修改数据
Alembic：修改和管理数据库表结构
```

### 迁移文件

```python
def upgrade() -> None:
    op.add_column(
        "users",
        sa.Column("email", sa.String(100), nullable=True),
    )


def downgrade() -> None:
    op.drop_column("users", "email")
```

### 常用命令

```bash
alembic revision --autogenerate -m "add email"
alembic upgrade head
alembic downgrade -1
alembic current
alembic history
```

### 注意事项

- 修改 SQLAlchemy 模型不会自动修改已有数据库。
- `--autogenerate` 生成后必须人工检查。
- 字段重命名可能被误判为“删除旧字段 + 新增字段”。
- 数据库结构能回退，不代表被删除的数据能恢复。
- 生产迁移应尽量向后兼容。

# 二、大模型生成与 Agent 基础

## 1. Top-k 与 Top-p

大模型生成每个 Token 时，会得到所有候选 Token 的概率分布。

### Top-k

只保留概率最高的 `k` 个 Token。

```text
top_k = 3
→ 只从概率最高的 3 个候选中采样
```

特点：候选数量固定。

### Top-p

按照概率从高到低累加，保留累计概率达到 `p` 的候选。

```text
top_p = 0.9
→ 保留累计概率覆盖约 90% 的候选
```

特点：候选数量动态变化。

### 核心区别

```text
Top-k：控制候选数量
Top-p：控制累计概率范围
```

---

## 2. 重新归一化与采样

假设裁剪后只剩：

```text
A：0.40
B：0.30
C：0.15
```

概率总和为 `0.85`，需要重新归一化：

```text
A：0.40 / 0.85 ≈ 0.471
B：0.30 / 0.85 ≈ 0.353
C：0.15 / 0.85 ≈ 0.176
```

之后按照新概率随机抽取一个 Token，这叫采样。

### 采样与贪心解码

- **采样**：高概率更容易选中，但低概率候选仍有机会。
- **贪心解码**：每次只选择概率最高的 Token。

## 3. 工具 Schema

### 定义

工具 Schema 是给大模型看的结构化接口说明，告诉模型：

- 工具名称
- 工具用途
- 参数名称
- 参数类型
- 必填字段
- 参数范围或枚举

```json
{
  "name": "search_docs",
  "description": "从工业知识库检索相关文档",
  "parameters": {
    "type": "object",
    "properties": {
      "query": {
        "type": "string",
        "description": "用户问题"
      },
      "top_k": {
        "type": "integer",
        "minimum": 1,
        "maximum": 20
      }
    },
    "required": ["query"]
  }
}
```

### Schema 与函数的区别

```text
Tool Schema：告诉模型怎么调用
Tool Function：真正完成查询、计算或操作
```

## 4. 工具函数是谁执行的

大模型通常不直接执行 Python 函数或系统命令。

```text
大模型：决定调用哪个工具并生成参数
Agent Runtime：校验并执行工具
工具函数：访问数据库、API、文件或 Shell
```

典型流程：

```text
用户问题
→ 模型生成 tool call
→ Runtime 找到对应函数
→ 执行函数
→ 把结果作为 ToolMessage 返回模型
→ 模型生成最终答案
```

只有工具函数内部使用 `subprocess`、Shell 或终端工具时，才会真正执行系统命令。

## 5. VLM 与 VLA

### VLM

**Vision-Language Model，视觉语言模型**。

```text
输入：图像/视频 + 文本
输出：文字回答、描述、识别结果
```

应用：

- 图像问答
- OCR 与文档理解
- 图表分析
- 工业缺陷描述
- 多模态对话

### VLA

**Vision-Language-Action Model，视觉语言动作模型**。

```text
输入：视觉信息 + 语言指令
输出：动作、轨迹、控制策略
```

应用：

- 机械臂抓取
- 机器人导航
- 自动驾驶
- 具身智能

### 核心区别

```text
VLM：能看、能理解、能说
VLA：能看、能理解、还能行动
```

## 7. Agent Runtime

### 定义

Agent Runtime 是让 Agent 真正运行起来的执行与调度系统。

### 主要职责

- 调用大模型
- 解析工具调用
- 执行工具
- 更新 Agent State
- 控制循环与终止
- 超时、重试和降级
- 权限和沙箱控制
- 检查点和持久化
- 日志与可观测性

### 最小 Agent Loop

```python
def run_agent(messages, tools):
    while True:
        response = llm.invoke(messages)
        messages.append(response)

        if not response.tool_calls:
            return response.content

        for call in response.tool_calls:
            tool = tools[call["name"]]
            result = tool(**call["args"])
            messages.append({
                "role": "tool",
                "tool_call_id": call["id"],
                "content": str(result),
            })
```

## 8. `with_structured_output()`

### 解决的问题

它主要解决：

```text
模型输出能否稳定转换为指定的数据结构
```

例如：

```python
from pydantic import BaseModel, Field
from typing import Literal


class RouteDecision(BaseModel):
    intent: Literal["explain", "sql", "rag"]
    topic: str
    confidence: float = Field(ge=0, le=1)
```

```python
structured_llm = llm.with_structured_output(RouteDecision)
```

它能约束：

- 字段是否存在
- 字段类型是否正确
- 枚举是否合法
- 数值范围是否合法
- 输出能否被程序解析

### 不能保证的问题

- 意图分类是否正确
- 内容是否符合事实
- 引用是否真的支持结论
- 模型自报置信度是否可信
- SQL 是否正确、安全
- 业务决策是否符合规则

### 核心结论

> `with_structured_output()` 提供的是 Schema 可靠性，不是事实可靠性。

## 10. `ChatPromptTemplate.from_template()` 与 `from_messages()`

### `from_template()`

默认创建一条 Human 消息：

```python
ChatPromptTemplate.from_template(
    "请解释 {topic}"
)
```

近似等价于：

```python
ChatPromptTemplate.from_messages([
    ("human", "请解释 {topic}")
])
```

### `from_messages()`

可以定义多种角色和历史占位符：

```python
from langchain_core.prompts import MessagesPlaceholder

prompt = ChatPromptTemplate.from_messages([
    ("system", "你是工业质量助手。"),
    MessagesPlaceholder("history"),
    ("human", "{question}")
])
```

### 核心区别

```text
from_template()：单条 Human 消息
from_messages()：多角色消息结构
```

# 三、RAG 评估指标

## 1. Ragas

Ragas 是用于评估 RAG、LLM 应用和 Agent 工作流的框架。

### 主要评估对象

```text
用户问题
→ 检索 Context
→ 模型生成 Response
```

### 四个核心指标

#### Context Precision

检索结果中有多少是真正相关的，且相关结果是否排在前面。

#### Context Recall

回答问题所需的关键信息是否都被检索出来。

#### Faithfulness

回答中的事实是否得到检索上下文支持。

#### Answer Relevancy

回答是否真正切中用户问题。

### 诊断逻辑

| 现象 | 可能问题 |
|---|---|
| Precision 低、Recall 高 | 找全了，但噪声太多 |
| Precision 高、Recall 低 | 找得准，但漏掉关键信息 |
| 检索好、Faithfulness 低 | 模型脱离上下文编造 |
| Faithfulness 高、Relevancy 低 | 没编造，但答非所问 |

---

## 2. MRR

**MRR = Mean Reciprocal Rank，平均倒数排名。**

它关注：

> 第一个正确结果排在第几位。

单个查询：

```text
RR = 1 / rank
```

例如：

```text
第 1 名正确 → 1.0
第 2 名正确 → 0.5
第 4 名正确 → 0.25
未找到       → 0
```

多条查询：

```text
MRR = 所有查询 RR 的平均值
```

### 与其他指标区别

| 指标 | 关注点 |
|---|---|
| MRR | 第一个正确结果排多前 |
| Precision@k | 前 k 个结果中有多少相关 |
| Recall@k | 相关结果被找回多少 |
| Hit Rate@k | 前 k 个结果中是否至少命中一个 |
| NDCG | 所有相关结果的整体排序质量 |

# 四、机器学习与特征工程

## 1. 标准化（Standardization）

### 定义

对每一列数值特征执行：

```text
z = (x - 均值) / 标准差
```

标准化后通常：

```text
均值约为 0
标准差约为 1
```

### 正确用法

```python
from sklearn.preprocessing import StandardScaler

scaler = StandardScaler()

X_train_scaled = scaler.fit_transform(X_train)
X_test_scaled = scaler.transform(X_test)
```

不能对测试集再次 `fit_transform()`，否则会造成：

- 训练集和测试集不在同一尺度
- 测试集信息泄漏

### 常见需要标准化的模型

- KNN
- K-Means
- SVM
- PCA
- 逻辑回归
- 神经网络

树模型通常不强制需要标准化。

### 异常值较多时

可以考虑：

```python
from sklearn.preprocessing import RobustScaler
```

---

## 2. 多重共线性

### 定义

多个自变量之间存在强线性关系，一个特征可以被其他特征近似线性表示。

例如：

```text
总面积 ≈ 客厅面积 + 卧室面积 + 厨房面积
```

### 影响

- 回归系数不稳定
- 系数方向可能异常
- 标准误变大
- 单个变量显著性下降
- 特征解释困难

### 检测方法

#### 相关系数矩阵

适合检查两两相关。

```python
corr = X.corr()
```

#### VIF

```text
VIF = 1 / (1 - R²)
```

经验判断：

```text
VIF ≈ 1：基本无共线性
VIF > 5：需要关注
VIF > 10：通常较严重
```

### 处理方式

- 删除重复或高度相关特征
- 总量与分量不要同时全部保留
- 使用 Ridge 或 Lasso
- 使用 PCA
- 结合业务意义迭代删除高 VIF 特征

### 注意

> 标准化不能消除多重共线性，只会改变数值尺度。

## 3. One-Hot 编码

类别特征经过 One-Hot 后，会拆成多个 0/1 列。

原始数据：

| 颜色 |
|---|
| 红色 |
| 蓝色 |
| 绿色 |

编码后：

| 颜色_红色 | 颜色_蓝色 | 颜色_绿色 |
|---:|---:|---:|
| 1 | 0 | 0 |
| 0 | 1 | 0 |
| 0 | 0 | 1 |

### sklearn 示例

```python
from sklearn.preprocessing import OneHotEncoder

encoder = OneHotEncoder(
    handle_unknown="ignore",
    sparse_output=False,
)

X_train_encoded = encoder.fit_transform(X_train[["颜色"]])
X_test_encoded = encoder.transform(X_test[["颜色"]])
```

### 线性模型中的虚拟变量陷阱

如果一个类别有 3 种取值，保留 3 个 One-Hot 列时：

```text
红色列 + 蓝色列 + 绿色列 = 1
```

这会形成完全共线性。

线性模型中可使用：

```python
OneHotEncoder(drop="first")
```

## 4. KNN 如何判断

KNN 的核心逻辑：

> 找到离新样本最近的 K 个训练样本，根据邻居进行预测。

### 分类

最近的 K 个样本进行投票：

```text
苹果：3 票
柠檬：2 票
→ 预测为苹果
```

### 回归

最近 K 个样本的目标值求平均：

```text
100、110、120
→ 预测为 110
```

### 为什么要标准化

KNN 基于距离。如果不同特征尺度差异巨大，大数值特征会主导距离。

```text
年龄：20～60
收入：50,000～1,000,000
```

不标准化时，收入几乎决定全部距离。

---

## 5. KNN 加权

普通 KNN：每个邻居权重相同。

```text
每个邻居都是 1 票
```

距离加权 KNN：距离越近，影响越大。

常见思想：

```text
权重 ≈ 1 / 距离
```

```python
from sklearn.neighbors import KNeighborsClassifier

model = KNeighborsClassifier(
    n_neighbors=5,
    weights="distance",
)
```

### 核心结论

> 加权就是让不同样本或特征对最终结果产生不同程度的影响。

## 1. TTL

**TTL = Time To Live，生存时间 / 有效期。**

### Redis

```bash
SET verification_code:1001 "836291" EX 300
```

表示缓存 300 秒后自动过期。

查看剩余时间：

```bash
TTL verification_code:1001
```

常见返回值：

```text
正整数：剩余秒数
-1：键存在，但没有过期时间
-2：键不存在或已过期
```

### 其他场景

| 场景 | TTL 含义 |
|---|---|
| Redis | 数据还能保存多久 |
| 缓存 | 结果还能使用多久 |
| DNS | DNS 结果还能缓存多久 |
| 消息队列 | 消息还能等待多久 |
| IP | 数据包还能经过多少跳 |

## 2. `sys.path.append(Path(...))`

正确写法通常是：

```python
import sys
from pathlib import Path

PROJECT_ROOT = Path(__file__).resolve().parents[1]
sys.path.insert(0, str(PROJECT_ROOT))
```

### 各部分含义

```text
__file__       当前 Python 文件路径
Path(...)      转换为 Path 对象
resolve()      获取规范绝对路径
parents[1]     获取上两级目录
sys.path       Python 模块搜索路径
insert(0, ...) 优先从项目根目录查找模块
```

### 常见错误

```python
Path(__file__).resolve().parent[1]
```

`parent` 是一个 Path 对象，不能用 `[1]`；应使用：

```python
.parents[1]
```

### 更规范方式

从项目根目录运行：

```bash
python -m scripts.test_llm
```

而不是：

```bash
python scripts/test_llm.py
```

正式项目还可以使用：

```bash
pip install -e .
```
